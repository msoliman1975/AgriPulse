1 The identity model

Identity is deliberately split across three stores. Understanding this split is the key to every flow and every gap below.

Keycloak (auth)
Platform Postgres (truth)
JWT (transport)
users — credentials, email, password. The login authority.
public.users — global person row, links to keycloak_subject.
sub / email / name — who you are.
groups tenant-<slug> — used for member discovery & legacy kcadm ops.
tenant_memberships + tenant_role_assignments — the authoritative role record. Supports many tenants per user.
tenant_id / tenant_role — single-valued attributes → claims.
realm roles — auto-created lazily on first assignment (TenantOwner, …).
platform_role_assignments + farm_scopes (farm-tier roles live only here + JWT).
platform_role / farm_scopes — drive RBAC in the API.
◆ How they connect at request time
Keycloak signs a JWT carrying tenant_id, tenant_role, platform_role and farm_scopes (projected from KC user attributes via protocol mappers). The API middleware (backend/app/shared/auth/middleware.py) validates the token against JWKS, builds a RequestContext (…/auth/context.py), then resolves permissions PlatformRole → TenantRole → FarmScope (first match wins) against app/shared/rbac/role_capabilities.yaml. Tenant routing does SET LOCAL search_path TO tenant_<uuid>, public — the tenant is taken from the claim, never the URL.

◆ Important asymmetry
Only platform and tenant roles map to Keycloak realm roles + attributes. The five farm-scoped roles exist only in the DB and the farm_scopes JWT claim — they are never realm roles. So the Keycloak realm only ever represents part of the authorisation model; the DB is the real source of truth.

3 What is seeded today

A fresh deploy seeds three things automatically. Note how little of it is identity data.

LayerSeeded byWhat lands
Keycloak realm infra/helm/keycloak/files/agripulse-realm.json (--import-realm, first boot only) The agripulse-api public client (+ 5 protocol mappers), the agripulse-tenancy service-account client (placeholder secret), and one user: dev@agripulse.local / password dev. No roles, no groups.
Realm post-install PostSync promote Job → promote-scripts/*.py Enables unmanagedAttributePolicy; sets platform_role=PlatformAdmin on the dev user; sets the tenancy client secret; grants realm-management roles to the service account; wires SMTP (Brevo); re-asserts the tenant mappers.
Platform Postgres Alembic migrations 0006…0027 Egyptian crops, imagery/weather catalogs, recommendation templates, platform setting defaults, grid thresholds, NDMI. No tenants, no users.
First PlatformAdmin Startup hook → platform_admins/bootstrap.py If PLATFORM_ADMIN_EMAIL is set and no PlatformAdmin exists yet, invites that user (idempotent). Default is empty → skipped.
Dev tenant + owner backend/scripts/dev_bootstrap.py (dev only) Creates dev-tenant schema and wires dev@agripulse.local as its TenantOwner. Not run in production.
▲ The net of it
Out of the box, the only usable login in any environment is the hard-coded dev@agripulse.local / dev seeded in the realm JSON (which ships in git and is identical for the dev and helm/prod realm files). Everything else — real platform admins, every tenant, every tenant user — is created at runtime through the API, and every runtime-created user depends on working SMTP to ever set a password. There is no production seed of a real admin or tenant.

4 Flow A — Fresh-deploy bootstrap

From helm install to a working first login.

  1. Keycloak pod
    Realm import on first boot
    StatefulSet starts with --import-realm, loads the realm JSON: two clients + the dev user. Import is first-time only.
  2. ArgoCD PostSync
    Promote Job runs (sync-wave 3)
    Waits for the admin REST to answer, then runs the three promote scripts: unmanaged-attributes ON, dev user → PlatformAdmin attribute + mapper, tenancy client secret set, service-account roles granted, SMTP configured.infra/helm/keycloak/templates/promote-bootstrap-job.yaml
  3. API pod
    Migrations + optional admin bootstrap
    Public-schema migrations seed catalogs/defaults. Lifespan hook reads PLATFORM_ADMIN_EMAIL; if set and no admin exists, calls invite_admin().app/core/app_factory.py · platform_admins/bootstrap.py
  4. Operator
    First login
    Signs in as dev@agripulse.local (or the bootstrapped admin, once they complete the emailed password reset). Token now carries platform_role=PlatformAdmin → full /platform access.
▲ Gap surfaced here
The env-var PLATFORM_ADMIN_EMAIL path calls invite_admin(), which creates the user password-less and fires an execute-actions-email reset. So even the "official" first admin cannot log in until Brevo SMTP works. The only SMTP-independent credential is the shipped dev/dev — which is unsuitable for production.

5 Flow B — Tenant creation + first owner

POST /api/v1/admin/tenants · requires platform.manage_tenants (PlatformAdmin only) · 3-step wizard at /platform/tenants/create.

  1. PlatformAdmin · FE
    Fills the wizard
    Profile (slug 3–32 [a-z0-9-], name, contact) → Owner (owner_email, owner_full_name, tier) → Review.frontend/src/modules/admin/pages/TenantCreatePage.tsx
  2. API
    Writes tenant rows + schema
    tenants, tenant_settings, tenant_subscriptions; then CREATE SCHEMA + per-tenant migrations.tenancy/service.py:372 · tenancy/bootstrap.py
  3. API → Keycloak
    Provisions the tenant group
    ensure_group(slug) → KC group tenant-<slug>, UUID stored on tenant.keycloak_group_id.
  4. API → Keycloak
    Invites the owner as TenantOwner
    invite_user() creates the KC user, adds to group, assigns the (auto-created) TenantOwner realm role, sets tenant_id+tenant_role attributes inline, fires the password-reset email. Mirrors DB rows in users, tenant_memberships, tenant_role_assignments.
  5. API
    Settles status
    KC ok → active, returns owner_user_id. KC failed → pending_provision (DB rows kept), retry via POST …/{id}/retry-provisioning.
  6. Owner
    Sets password & logs in
    Receives reset email → sets password → first login lazily confirms membership; token carries their tenant_id.
Tenant lifecycle state machine

active ⇄ suspended → pending_delete (30-day grace, cancel_delete → suspended) → purge (irreversible). Failed provisioning → pending_provision → retry_provisioning → active. Suspended/pending-delete tenants block non-platform logins at the middleware.

6 Flow C — User invitation & login

POST /api/v1/users:invite · requires user.invite (TenantOwner / TenantAdmin) · UI at /settings → Users. There is no public self-registration — the realm sign-up form is disabled on purpose.

  1. TenantAdmin · FE
    Invites by email + role
    Role ∈ {TenantOwner, TenantAdmin, BillingAdmin, Viewer}. Farm-scoped roles are granted separately, per farm.
  2. API
    Reuse-or-create the global person
    One public.users row per email across all tenants; keycloak_subject holds the KC UUID, or pending::<email> if KC is unavailable.iam/users_service.py:174
  3. API → Keycloak
    Create user, group, role, attributes
    User + group membership + realm role + tenant_id/tenant_role attributes set inline in the POST (KC26 PUT-wipes-missing-fields gotcha).
  4. API → Keycloak
    Send password-reset email
    PUT /users/{id}/execute-actions-email with UPDATE_PASSWORD. Best-effort — logged, never fails the invite.
  5. API
    Mirror DB rows + audit
    tenant_memberships + tenant_role_assignments; iam.user_invited audit event.
  6. User
    Reset → login
    Email link → set password → login. Within ≤15 min the token reflects their role.

9 Recommended seeding strategy

Concrete changes — ordered by leverage — to make the system seed-driven and cut Keycloak hand-management to near zero.

✓ S1 — Seed realm roles & remove lazy auto-create (fixes G5)
Add a roles.realm[] block to agripulse-realm.json listing all platform + tenant roles (TenantOwner, TenantAdmin, BillingAdmin, Viewer, PlatformAdmin, PlatformSupport). Then change _assign_realm_role() to look up only (error on missing) instead of auto-creating — so a typo fails loudly instead of inventing a role.
✓ S2 — Replace dev/dev with environment-gated seed users (fixes G1, G4)
Split the realm file: keep dev@agripulse.local / dev only in infra/dev/keycloak/. For helm/prod, seed a real bootstrap admin whose credential is a temporary password ("temporary": true) pulled from a Secret/ExternalSecret, forcing UPDATE_PASSWORD at first login — no SMTP needed. Set bruteForceProtected: true for non-dev.
✓ S3 — SMTP-independent invite fallback (fixes G2)
When execute-actions-email fails (or always, behind a flag), have invite_user() set a one-time temporary password and return a reset link / token in the API response. Surface a "Copy invite link" + "Resend email" action in the Users UI so an admin can hand off credentials out-of-band. This removes the single hard SMTP dependency from the whole onboarding chain.
✓ S4 — Make provisioning the default & fail-loud (fixes G3)
Flip keycloak_provisioning_enabled to True in the helm values (it already needs the tenancy secret, which the promote job sets). Add a startup readiness check that logs a prominent warning if the Noop client is wired in a non-dev profile, so a misconfigured cluster is obvious rather than silently no-op.
✓ S5 — Optional production tenant seed (fixes G4)
Promote dev_bootstrap.py's tenant+owner logic into a guarded, idempotent seed job driven by env/Secret (SEED_TENANT_SLUG, SEED_OWNER_EMAIL). Lets a new environment come up with one real tenant + owner with zero clicks and zero Keycloak console time.
✓ S6 — DB→Keycloak reconcile job (fixes G6, G11)
A periodic (or post-write) reconciler that reads tenant_role_assignments / membership status as truth and pushes attribute + enabled-flag updates into Keycloak. Closes the drift window and ensures DB soft-deletes actually disable the login.
✓ S7 — Multi-valued tenant claim + switcher (fixes G7)
Make tenant_role/tenant_id multivalued (or add a memberships claim) and add a tenant-switcher in the UI, so users who legitimately belong to multiple tenants aren't pinned to one. Larger change — schedule behind the above.

Priority order

1 · S2 + S3 (unblock prod login) 2 · S4 (provisioning on) 3 · S1 (seed roles) 4 · S5 (tenant seed) 5 · S6 (reconcile) 6 · S7 (multi-tenant)

◆ Net effect on Keycloak management
With S1–S5 in place, the only standing Keycloak chores left are key-rotation restarts (G9) and intentional realm-shape edits (G8) — everything about tenants, users, and roles becomes seed- or API-driven, and the Keycloak admin console drops out of the day-to-day operating loop.
