# CD-11 — GitHub OIDC role for Terraform apply + image push

[Shared preamble — see README.md]

## Goal
Replace any static AWS credentials in GitHub Actions with OIDC federation. CI assumes a scoped IAM role at runtime — no `AWS_ACCESS_KEY_ID` secrets in repo settings. Also adds a new CI job that runs `terraform apply` on merge to main (manual environment approval).

## Files to change
- `infra/terraform/github-oidc.tf` — new. OIDC provider for `token.actions.githubusercontent.com` + two roles:
  - `gha-terraform-plan` (read-only on most services + Terraform state RW). Assumed on PRs.
  - `gha-terraform-apply` (full write on services Terraform manages). Assumed on push to main with environment approval.
- `infra/terraform/iam-policies/` — new directory with the policy JSON files referenced above.
- `.github/workflows/ci.yml` — extend the `infra-tf` job to also run `terraform plan` against real AWS (using the plan role) and post the plan as a PR comment.
- `.github/workflows/infra-apply.yml` — new. Triggers on push to main when `infra/terraform/**` changed. Uses environment `production-infra` requiring manual approval.
- `.github/workflows/ci.yml` — the `containers` job already pushes to GHCR; no change needed there. **Don't** add ECR push.

## Tasks
1. Terraform:
   - `aws_iam_openid_connect_provider` with thumbprint pinned to GitHub's current value (refresh quarterly).
   - Plan role: trust policy restricted to `repo:<owner>/<repo>:pull_request`. Permissions: `*:Get*`, `*:List*`, `*:Describe*` plus Terraform state lock RW on the DynamoDB lock table.
   - Apply role: trust policy restricted to `repo:<owner>/<repo>:ref:refs/heads/main`. Permissions: scoped to the resource types Terraform manages (EKS, VPC, IAM, KMS, S3, Route 53, ACM, Secrets Manager, EventBridge, SQS). NOT `*:*`.
2. `ci.yml` plan job:
   - Add `permissions: id-token: write` at job level.
   - Use `aws-actions/configure-aws-credentials@v4` with `role-to-assume: <gha-terraform-plan ARN>`.
   - Run `terraform plan -no-color -out=plan.tfplan`.
   - Post plan output as a PR comment (use `marocchino/sticky-pull-request-comment@v2`).
3. `infra-apply.yml`:
   - Trigger: `push` to `main`, paths `infra/terraform/**`.
   - Environment: `production-infra` (requires manual reviewer approval before the job runs).
   - Assume `gha-terraform-apply` role.
   - `terraform init && terraform apply -auto-approve`.
   - On failure, post a Slack notification (use Slack webhook from repo secret — keep this one static, it's not AWS).
4. Migrate any existing `AWS_*` secrets out of repo settings if they exist. Document removal in PR description.

## Out of scope
- Don't extend OIDC to ECR (we're staying on GHCR).
- Don't add SOPS or git-crypt for Terraform variables. Sensitive variables come from SM via `data.aws_secretsmanager_secret_version` references in Terraform.
- Don't add a separate state-import workflow.

## Definition of done
- `terraform apply` from CD-10 (which presumably ran from your laptop) cleanly transitions to the OIDC roles — re-running with the OIDC role assumed produces no diff.
- A test PR with a no-op Terraform comment change triggers a plan + PR comment.
- A merge to main with a real Terraform change triggers `infra-apply.yml`, waits for approval, applies cleanly.
- No `AWS_ACCESS_KEY_ID` secrets in repo settings.
